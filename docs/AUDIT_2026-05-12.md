# Prime Cars — Code Audit

**Date:** 2026-05-12
**Scope:** Full repository review — configs, App Router boundaries, auth/RBAC, state and data fetching, i18n consistency, lint/type-check.
**Method:** Read configs and representative files, run `tsc` and `eslint`, count anti-patterns programmatically.

This file is a punch list. Each finding has a file path and (where useful) line numbers so it can be verified directly. Items are ordered by severity, not by location.

---

## Severity 1 — Broken or security-relevant

These items are not style preferences. They are real bugs or real security exposure.

### 1.1 No server-side auth on admin routes

- **Where:** `src/middleware.ts`, `src/app/[locale]/admin/layout.tsx`
- **Evidence:** `middleware.ts` only configures `next-intl`'s `createMiddleware` for locale routing. There is no auth check. Every `/admin/*` page is a Server Component that renders `<AdminLayout>` (a Client Component) which calls `useUser()` and conditionally shows a `<LoginModal>`.
- **Impact:**
  - Admin route HTML and the entire admin JS bundle are sent to every visitor, authenticated or not.
  - The "protection" is purely client-side cosmetic. The login modal is bypassable; a curious user can read endpoint names, internal route structure, and React component tree from `view-source`.
  - The backend presumably rejects unauthorized requests, so this isn't direct account compromise — but the admin surface area is leaked.
- **Note:** `src/components/rbac/ProtectedRoute.tsx` was written but is **not used** anywhere in the admin layout.
- **Fix:** Move auth check into `middleware.ts` (reading the session cookie or token on each request) and/or use a Server Component layout that calls `notFound()` / `redirect()` based on a server-readable session.

### 1.2 Refresh-token lock leaks pending promises on failure

- **Where:** `src/lib/auth/token.ts:80-84` and `:100-132`
- **Evidence:** When a refresh is in flight, concurrent 401s call `addRefreshSubscriber` and return a `Promise` that will be resolved via `onTokenRefreshed`. On the failure paths (`!response.ok` and `catch`), `removeTokens()` is called which sets `refreshSubscribers = []` — but the subscribers' promise resolvers are never called. Those awaiters hang forever.
- **Impact:** Memory leak plus dangling network requests during refresh failures. In practice you see "loading forever" UX after a session expires under load.
- **Fix:** Track success and failure subscribers separately, or change the contract so subscribers receive `string | null` and handle the null path.

### 1.3 Refresh token stored in `localStorage`

- **Where:** `src/lib/auth/token.ts:41-43`
- **Evidence:** Both access and refresh tokens are written to `localStorage`.
- **Impact:** Any XSS — including a compromised npm dependency — exfiltrates both tokens, giving an attacker effectively permanent access via refresh.
- **Fix:** Server should set refresh token as an HttpOnly, Secure, SameSite=Lax (or Strict) cookie. Frontend only handles the access token. If backend can't change, at minimum move the refresh token to `sessionStorage` (lifetime-bounded) and accept that XSS is still fatal but reduces window.

### 1.4 Module-singleton `QueryClient`

- **Where:** `src/lib/react-query/client.ts:7`, `src/contexts/QueryProvider.tsx`
- **Evidence:** `new QueryClient(...)` is created at module load time and imported by the provider, instead of being instantiated inside the provider via `useState(() => ...)`.
- **Impact:** Canonical Next.js App Router footgun. If anything ever renders this provider on the server (RSC + Client island composition can trigger this), the cache is shared between concurrent users' requests on the same server process. Data leak.
- **Fix (mechanical):**
  ```tsx
  // QueryProvider.tsx
  const [client] = useState(() => new QueryClient({ /* opts */ }));
  return <QueryClientProvider client={client}>{children}…</QueryClientProvider>;
  ```

### 1.5 Subscriber-list race on concurrent refresh failures

- **Where:** `src/lib/auth/token.ts` (related to 1.2)
- **Evidence:** `refreshSubscribers = []` is reassigned during failure handling and during `onTokenRefreshed`. Concurrent failure + success or two failures racing can drop subscribers.
- **Fix:** Snapshot the array, then clear, then iterate the snapshot.

## Severity 2 — Architecture wrong (not a bug, just wrong shape)

### 2.1 App Router used as an SPA shell

- **Where:** `src/app/[locale]/cars/page.tsx`, `src/components/pages/CarsPage.tsx`, `src/components/pages/CarDetailsPage.tsx`
- **Evidence:** Server Component pages are one-liners that render a `"use client"` 500+ line page component that fetches data via TanStack Query in the browser.
- **Counterexample (correct pattern in this repo):** `src/components/pages/HomePage.tsx` is an async Server Component that calls `fetchAllAvailableCars()` and passes data down.
- **Impact:** No SEO benefit for the cars listing (the cars are not in initial HTML), no streaming, larger client bundles, redundant request waterfalls.
- **Fix:** Convert `/cars` and `/cars/[id]` to Server Components that fetch the data; pass it into a small `"use client"` island that handles only the interactive parts (filters, sort, pagination).

### 2.2 `reactStrictMode` declared twice in `next.config.js`

- **Where:** `next.config.js:5` (`false`) and `:17` (`true`)
- **Evidence:** Object literal has the key twice. Last one wins (`true`), but this means someone disabled it and someone re-enabled it without removing the old line.
- **Fix:** Delete one. Keep `reactStrictMode: true`.

### 2.3 Duplicate `<meta name="viewport">`

- **Where:** `src/app/[locale]/layout.tsx:39-42` and `:100`
- **Evidence:** Both `export const viewport: Viewport` (the App Router idiom) and a manually-rendered `<meta name="viewport" content="…" />` inside `<head>` exist. The rendered HTML has two viewport tags.
- **Fix:** Delete the manual `<meta>` tag.

### 2.4 `className="dark"` hardcoded on `<html>`

- **Where:** `src/app/[locale]/layout.tsx:98`
- **Evidence:** Initial server-rendered HTML always says `<html class="dark">`. `ThemeInitScript` then patches it from `localStorage`.
- **Impact:** Flash of dark content (FOUC) for light-mode visitors before the init script runs.
- **Fix:** The init script should write the class onto `<html>` before paint via inline `document.documentElement.classList.add(saved)`, and the HTML element should not pre-bias the class.

### 2.5 Mega-components

- **Where:** `src/components/ImportCalculator.tsx` (1048 lines, 33 `useState` calls in the first 100 lines), `src/components/admin/modals/UpdateCarModal.tsx` (910), `src/components/admin/views/SettingsView.tsx` (786), `src/components/admin/modals/AddCarModal.tsx` (740).
- **Evidence:** `ImportCalculator.tsx:70-102` is 33 `useState` calls in a row — each piece of state independent of the others.
- **Impact:** Impossible states are representable (`showResults=true` with `calculationResults=null`); changes to any one field re-render the entire 1000-line tree; component is unreviewable.
- **Fix:** Replace the cluster of `useState`s with a `useReducer` or a real state machine (XState or hand-rolled). For modals: split the form into named sub-components, drive the data layer with RHF + Zod (the project already has both).

### 2.6 122 `"use client"` files, 0 `"use server"` files

- **Evidence:** `grep -rl '"use client"' src | wc -l` → 122. `grep -rl '"use server"' src | wc -l` → 0.
- **Impact:** Every mutation flows through `authenticatedFetch` from a Client Component. Server Actions — the canonical App Router mutation path — aren't used at all. This forces TanStack Query mutations everywhere and pushes auth-state complexity to the client.
- **Fix:** Migrate at least the high-value admin mutations (create/update/delete car, user) to Server Actions. Backend already authenticates, but Server Actions remove a layer of client-side bookkeeping and let you `revalidatePath`/`revalidateTag` instead of TanStack invalidation.

### 2.7 `useEffect` for derived state, pervasively

- **Total:** 98 `useEffect` calls across the codebase.
- **Concrete example:** `src/hooks/useCarsPage.ts:38-48` — uses `useEffect` + `useRef` to shuffle the cars list exactly once after data arrives. Should be `useMemo` keyed on data identity, or sorted at the data layer.
- **Concrete example:** `src/components/admin/pages/AdminCarsPage.tsx:41-46` — `useEffect` calls `state.loadCars()` when `user` becomes available, suppressed with `eslint-disable-next-line react-hooks/exhaustive-deps`. Should be `useQuery({ enabled: !!user })`.
- **Fix:** Audit every `useEffect`; if it's not syncing to an external system (DOM, browser API, websocket, etc.), it's almost certainly wrong.

### 2.8 Vestigial API surface in `useCarsPage`

- **Where:** `src/hooks/useCarsPage.ts:65-71`
- **Evidence:**
  ```ts
  // eslint-disable-next-line @typescript-eslint/no-unused-vars
  const isLoading = (_category?: string) => isLoadingAll;
  // eslint-disable-next-line @typescript-eslint/no-unused-vars
  const loadCarsForCategory = async (_category?: string) => refetch();
  ```
  Both functions ignore their parameter. Leftover from a refactor that used to support per-category loading.
- **Fix:** Remove. Update callers to use `isLoadingAll` / `refetch` directly.

### 2.9 Two context directories

- **Where:** `src/contexts/` (UserContext, QueryProvider) and `src/components/ThemeContext.tsx`.
- **Impact:** Inconsistent. Theme lives next to buttons.
- **Fix:** Move `ThemeContext.tsx` into `src/contexts/`.

## Severity 3 — Quality and bundle

### 3.1 Dead form library still depended on

- **Where:** `package.json:50` (`formik`), `:72` (`zod-formik-adapter`)
- **Evidence:** `grep -rn 'from "formik"' src` returns nothing. Both packages are orphaned.
- **Fix:** `npm rm formik zod-formik-adapter`. The CLAUDE.md explicitly notes this should already be done.

### 3.2 `next-themes` in deps but custom `ThemeContext` is used

- **Where:** `package.json:58`
- **Evidence:** Only `src/app/globals.css` mentions `next-themes` (CSS selector, not an import).
- **Fix:** `npm rm next-themes`.

### 3.3 Two competing Radix layers

- **Where:** `package.json:43` (`@radix-ui/themes`) and `:17-42` (~25 individual `@radix-ui/react-*` primitives).
- **Evidence:** `@radix-ui/themes` is only imported in 7 admin files (`admin/layout.tsx` wraps content in `<Theme appearance="inherit">`; admin views/primitives import `<Box>`, `<Flex>`, etc.). The individual `@radix-ui/react-*` primitives are used everywhere else via shadcn-style components in `src/components/ui/`.
- **Impact:** Two design systems shipped to the client, two sets of primitives bundled.
- **Fix:** Pick one. Either commit to Radix Themes (drop the shadcn-style `ui/` directory and the individual `@radix-ui/react-*` imports) or drop `@radix-ui/themes` and rewrite the 7 admin files to use the existing `ui/` primitives. Recommend the latter — your design system already lives in `ui/`.

### 3.4 Two icon libraries

- **Where:** `package.json:53` (`lucide-react`) and `:63` (`react-icons`)
- **Fix:** Pick one. Lucide is already used throughout.

### 3.5 ESLint config is bare minimum

- **Where:** `.eslintrc.json`
- **Evidence:** Two lines: `next/core-web-vitals` + `next/typescript`. Lint run: 0 errors, 1 warning across 275 files. The code isn't clean — nothing is being checked.
- **Fix:** Add `eslint-plugin-react-hooks` with `exhaustive-deps` set to `error`, `eslint-plugin-jsx-a11y` (recommended), `eslint-plugin-import` for ordering, optionally `eslint-plugin-tailwindcss`. Optionally `eslint-plugin-no-restricted-imports` to ban `next-themes`, `formik`, raw `fetch` to API_BASE_URL.

### 3.6 TypeScript strictness is intro level

- **Where:** `tsconfig.json`
- **Evidence:** Only `"strict": true`. Missing:
  - `noUncheckedIndexedAccess` — would catch many bugs around array/record access
  - `exactOptionalPropertyTypes` — distinguishes `undefined` vs missing
  - `noImplicitOverride`
  - `noUnusedLocals` / `noUnusedParameters`
  - `noFallthroughCasesInSwitch`
- **Fix:** Turn on `noUncheckedIndexedAccess` first; budget time to fix the resulting errors. Then progressively enable the others.

### 3.7 Two `tsconfig` files

- **Where:** `tsconfig.json`, `tsconfig.lint.json`, `package.json:12` (`type-check` uses the lint config).
- **Evidence:** `tsconfig.lint.json` just extends the main config and adds `.next`/`build` to excludes. No real divergence.
- **Fix:** Delete `tsconfig.lint.json`, run `tsc --noEmit --project tsconfig.json`.

### 3.8 39 `console.*` calls in source

- **Where:** various
- **Impact:** Ships to production. Includes error logs that should be structured.
- **Fix:** Replace with a small logger module, or enable Next's compiler `removeConsole` option in production. Audit `console.error` usages first — some carry useful state that should go to a real observability sink.

### 3.9 15 `eslint-disable` directives

- **Evidence:** `grep -rn 'eslint-disable' src` shows 15 hits.
- **Fix:** Each one is a paved-over decision. Audit. Most are `react-hooks/exhaustive-deps` masking effect anti-patterns covered in 2.7.

### 3.10 `compress: true` in `next.config.js:19`

- **Impact:** Vercel handles compression at the edge. Setting this duplicates work on the Node process.
- **Fix:** Delete.

### 3.11 `Permissions-Policy: ... interest-cohort=()`

- **Where:** `next.config.js:34`
- **Evidence:** FLoC was abandoned years ago.
- **Fix:** Remove the `interest-cohort` directive (harmless, but cargo).

- **Fix priority order:**
  1. Add `vitest` (~5 min setup).

  4. Component tests with `@testing-library/react` for the form-heavy admin modals.

- **Impact:** The root CLAUDE.md is 18kB — too coarse and at risk of being ignored. Anthropic's documented practice is per-directory CLAUDE.md files for sub-area conventions.
- **Fix:** Move admin-specific conventions to `src/components/admin/CLAUDE.md`. Move data-fetching conventions to `src/lib/CLAUDE.md`. Keep root short.

## What this audit does NOT cover

These would need a separate, deeper pass:

- **Accessibility** beyond what ESLint with `jsx-a11y` would surface. The codebase uses Radix primitives which are accessible by default, but a full keyboard-and-screen-reader pass would surface real issues.
- **Performance profiling.** No bundle analysis was run. Given the dep bloat (4 UI systems, 2 icon libraries), the bundles are almost certainly larger than necessary, but the exact numbers aren't measured here.
- **SEO beyond metadata.** The metadata setup looks reasonable; structured data (JSON-LD) exists. But the cars listing being client-side-fetched is a real SEO miss.
- **Backend contract.** Every backend response is assumed to be `{ data: ... }` — actual schema verification (with Zod parsing at the network boundary) is missing and would catch real bugs.
- **Image optimization.** `next.config.js` has reasonable `remotePatterns`, but the actual image usage in components wasn't audited for sizes/priority.
